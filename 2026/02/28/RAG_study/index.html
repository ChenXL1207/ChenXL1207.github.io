2. RAG基本概念

2.1 什么是RAG？

RAG（检索增强生成）是一种先从外部知识库检索相关信息，再把这些信息交给大模型生成回答的技术，能让大模型不编造内容、用上最新知识、只回答指定领域内容，从根本解决大模型幻觉、知识过时、隐私数据无法使用的问题。

2.2 RAG的核心优势

  • 实时获取最新信息，不受模型训练时间限制
  • 降低**幻觉（Hallucination）**问题
  • 支持自定义数据库
  • 提高回答的准确性和可靠

3. LangChain介绍

LangChain 是一套以 Python/JavaScript 为核心的开源框架，它并非直接提供 AI 能力，而是像 “AI 应用的乐高积木” 一样，将大模型（如 GPT、文心一言）、外部数据源（文档 / 数据库）、工具（搜索引擎 / 代码解释器）、向量数据库（Chroma/Milvus）等 AI 应用开发的核心组件封装成标准化接口和可组合的 “链（Chain）”，让开发者无需从零编写所有逻辑，就能快速搭建包含检索增强生成（RAG）、智能体（Agent）、多轮对话等复杂能力的 AI 应用，大幅降低 AI 应用从原型到落地的开发成本和复杂度。

3.2 核心模块-模型

模型在表现方向上包括两大类型

  • 语言模型
  • 文本嵌入模型，其中文本嵌入模型，个人理解：将文本与向量结合起来，可使用余弦相似度算法分析其表现意思是否唯一或类似

3.2.1 语言模型

语言模型，它有两种不同的子类型：LLMs 和 Chat Models。LLMs 封装了接受文本输入并返回文本输出的 API，而 Chat Models 封装了接受聊天消息输入并返回聊天消息输出的模型。尽管它们之间存在细微差别，但使用它们的接口是相同的。我们可以导入这两个类，实例化它们，然后在这两个类上使用 predict 函数并观察它们之间的区别。但是，您可能不会直接将文本传递给模型，而是使用提示（prompts）。

3.2.2 文本嵌入模型

文本嵌入模型（Text Embedding Model）是一类将文本（单词、句子、段落）转换成高维数值向量的 AI 模型，核心原理是通过深度学习捕捉文本的语义信息 —— 语义相似的文本会生成距离更近的向量，语义差异大的文本向量距离更远，比如 “苹果手机” 和 “iPhone” 的向量高度相似，而和 “香蕉” 的向量差异显著。这种将非结构化文本转为结构化数值的过程，解决了计算机无法直接理解自然语言的问题，是实现语义检索、文本聚类、相似度计算等任务的基础。

图片来自黑马程序员B站

在 RAG 等 AI 应用中，嵌入模型是连接文本和向量数据库的核心桥梁：一方面，它把外部知识库的文本块转换成向量存入数据库；另一方面，它把用户的提问也转换成向量，通过向量间的相似性检索，快速找到和问题语义匹配的文本内容，这也是 RAG 能实现 “精准检索” 的关键前提。常见的嵌入模型既有 OpenAI 的 text-embedding 系列、阿里云的通义嵌入等商用 API 模型，也有 BGE、m3e 等免费开源的本地化部署模型。

3.4 核心模块-索引（indexs）

LangChain 中的 “索引（Index）” 并非传统数据库的索引，而是对外部文本数据的 “结构化封装” —— 它将非结构化的文本（PDF / 文档 / 网页）处理成 “可快速检索” 的格式（如向量索引、关键词索引），并封装成统一的Retriever接口，让你能一键检索和问题相关的文本内容。

3.4.2 文本分割器

文本分割器是 LangChain 中处理长文本的核心工具，核心作用是将大段的非结构化文本（比如网页、PDF、长文档）分割成小的 “文本块（Chunk）”，适配嵌入模型 / 大模型的上下文长度限制。

LangChain 中常用的文本分割器（按场景选）

  • RecursiveCharacterTextSplitter（主流）
    • 按 “分隔符优先级” 递归分割（先按大分隔符，再按小分隔符）,适用中文文本、通用长文本（网页 / PDF）
    • 优点：适配中文语义，避免断句；缺点：需自定义分隔符
  • CharacterTextSplitter（基础）
    • 按固定字符数分割，不考虑语义，适用无结构的纯文本（如日志）
    • 优点：简单；缺点：易切断语义（比如把一句话拆成两半）
  • TokenTextSplitter（精准）
    • 按 token 数分割（适配模型 token 限制），适用对 token 敏感的场景（如 GPT 系列）
    • 优点：精准匹配模型限制；缺点：需计算 token，稍复杂
  • MarkdownTextSplitter（专用）
    • 按 Markdown 格式分割（标题 / 列表 / 代码块），适用Markdown 文档、笔记
    • 优点：保留格式语义；缺点：仅适配 MD 文件

3.5 核心模块-内存（Memory）

Memory 是 LangChain 中管理对话上下文的模块，核心作用是 “保存历史对话记录”，让大模型能基于 “当前问题 + 历史对话” 生成回答，而非每次都 “失忆”（仅回答当前问题）

  • ConversationBufferMemory（基础）
    • 按顺序保存所有对话记录（纯文本）
    • 简单多轮对话（如客服问答）
  • ConversationBufferWindowMemory（窗口）
    • 只保存最近 N 轮对话（避免上下文过长）
    • 长对话场景（如多轮咨询）
  • ConversationSummaryMemory（摘要）
    • 用大模型总结历史对话（压缩上下文）
    • 超长对话（如几十轮聊天）
  • VectorStoreRetrieverMemory（语义）
    • 将历史对话存入向量库，检索相关记忆
    • 复杂语义对话（如 RAG + 多轮问答）

3.6 核心模块-链路（Chains ）

Chains 是 LangChain 中串联多个组件的 “流水线”，核心作用是将 “PromptTemplate、LLM、Memory、Retriever” 等组件按逻辑组合，实现复杂任务（而非单一的 “调用模型”）。

  • 没有 Chains：你需要手动写代码拼接 Prompt、调用模型、处理 Memory，代码冗余且易出错；

  • 有 Chains：只需定义组件和流程，LangChain 自动处理 “输入→组件 1→组件 2→输出” 的全流程。

常用Chain类型

  • LLMChain（基础）
    • 仅串联 “PromptTemplate + LLM”
    • 单轮问答、简单生成（如文本总结）
  • ConversationChain（对话）
    • 串联 “LLM + Memory”
    • 多轮对话（如上面的 Memory Demo）
  • RetrievalQA（RAG 核心）
    • 串联 “Retriever + LLM”
    • 基于知识库的问答（如网页文本问答）
  • SequentialChain（串联）
    • 按顺序执行多个 Chain（输出作为下一个输入）
    • 复杂任务（如 “文本总结→关键词提取→回答问题”）

3.7 核心模块-代理（Agents ）

Agents（代理）是 LangChain 中最高阶的核心模块，本质是 “能自主决策的 AI 助手”—— 它不再像 Chains 那样按人工预设的固定流程执行任务，而是让大模型根据用户的目标，自主判断需要调用哪些工具、执行哪些步骤，最终完成复杂任务。

step 1：文档加载与分割

step 3：存储与索引

step 4：LLM调用

基于检索结果生成最终答案。
